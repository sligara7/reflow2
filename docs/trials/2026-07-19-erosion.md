# Erosion — after N fix cycles, does the design describe what shipped?

**What:** a clean, coherent thread (requirement → capability → component → artifact → verification)
put through five rounds of *test fails → fix the code → accept the drift*, then released. Driven
over the real MCP binary by [`tools/erosion_trial.py`](../../tools/erosion_trial.py).

**Why — a correction to the [phase-coverage trial](2026-07-19-phase-coverage.md).** That trial
scored P3 **4/4** and the score was misleading. It measured whether the graph notices a *drift
event*: one file, changed once, detected once. The failure that actually sank the original reflow is
not an event. It is the cycle:

> write code → test → fix → test → fix → … → cycle N → test → **release**

where every individual step is legitimate — a test failed, someone fixed the code — and **nobody
ever decides to diverge from the design.** After N iterations the code is the truth and the design
is fiction, and no single moment was wrong. Detecting "this file changed" is nearly worthless
against that, because the answer is always *"yes, I know, I fixed a bug."* The question is not
whether a file moved. It is whether, at release, **the design still describes what was released.**

**Setup.** `cap:charge` — *"Charges once per idempotency key, rejecting duplicates within 24h."*
Five fix cycles, of which the fourth quietly widens the dedup window from 24h to 7 days: a real
behaviour change, of exactly the kind a failing test produces, that nobody writes back into the
design. Then ship it.

## Result — 2 of 7

```
NO   the design flags that the code moved under it            gaps: []
YES  the capability description still says 24h (now false)
NO   anything flags a description contradicted by the code it points at
YES  the graph records that this artifact drifted             1 DriftEvent (of 5)
NO   the release records WHAT it contains
NO   you can ask "does what shipped match what was designed?"
NO   verification coverage distinguishes passing from merely present
```

---

## 1. After full erosion and a release, the graph reports **zero gaps**

That is the headline and it is worse than any individual miss. `detect_gaps` returns `[]`. The
design describes a system that rejects duplicates within 24 hours; the deployed system uses 7 days;
the requirement behind it is about not double-billing, which the change materially affects. And the
graph reports perfect coherence.

Every drift was detected. Every drift was accepted. Coherence, as currently measured, survived all
five — because what is measured is whether the bookkeeping is *complete*, never whether it is *true*.

## 2. Accepting drift is one-sided — this is the erosion mechanism

Each cycle ends with `set_artifact_checksum`, which the smoke test calls *"an accepted change is the
new baseline."* It updates the code-side baseline and **asks nothing about the design.**

That makes accept locally reasonable and globally fatal. The developer's fix *was* correct; the test
*does* now pass. Nothing in the loop ever poses the second half of the question:

> the code moved — should the **design** move too, or was the code wrong?

Accepting drift as it stands is neither of those answers. It is *"stop telling me."* Run it N times
and you have the reflow1 outcome exactly, with every step defensible.

The right shape is that accepting drift is a **two-sided decision**, and the design records which
way it went: the capability description is updated to match what was built, or the divergence is
marked as a defect in the code. A third option — accept the file, leave the design alone, say
nothing — should not exist, because it is the one that erodes.

## 3. The drift record overwrites itself, erasing the accumulation

Five drifts on one artifact leave **one** `DriftEvent`. The id is derived with no time or sequence
component (`drift.rs`):

```rust
fn drift_event_id(artifact_id: &str, kind: DriftKind) -> String {
    format!("drift:{:016x}", fnv1a(&format!("{}|{}", kind.as_str(), artifact_id)))
}
```

so every `checksum_change` on the same artifact writes the same node id and replaces the previous
one. The single signal that would reveal erosion — *this artifact has drifted five times and its
capability was never revisited* — deletes itself on each new occurrence, leaving only the most
recent.

This contradicts the project's own axis-Z invariant, stated in AGENTS.md as **CHANGE — never
overwrite the past**, which `temporal.rs` implements with epochs and snapshots for design edits. The
as-built side has no such discipline. "Drifted once" and "drifted twenty times" are indistinguishable,
and the second is the one that means the design is gone.

## 4. There is no as-released view, and no vocabulary for one

Checked two ways: **`DEPLOYED_TO` (Release → Environment) is the only edge in the schema involving
`Release`.** Nothing links a Release to the Artifacts or Components it shipped — even though
`Release`'s own extraction hint reads *"A packaged, operable version of some Components/Artifacts."*
The intent is written in prose with no edge to carry it.

So "does what we released match what we designed?" is not an unimplemented query. It is a question
the graph **has no way to express**. reflow2 has an as-designed view and a partial as-built view, and
the third — as-released, the one the user actually lives with — has no structure at all.

This is more than [BL-9](../backlog.md)'s `reconcile_deployment`: a reconciler needs something to
reconcile, and the containment edge has to exist first.

## What this changes

The [phase-coverage trial](2026-07-19-phase-coverage.md)'s conclusion stands but its P3 score should
be read as **4/4 at detecting events, 0/2 at retaining coherence across them.** Both trials point the
same way and this one sharpens it:

> The golden thread works. Reality is fed into it once, at P3, as an event —
> and there is no mechanism anywhere that obliges the design to move when the
> implementation does.

Raised: **BL-33** (accepting drift is one-sided; the drift record overwrites itself), **BL-34**
(no as-released view — `Release` has no containment edge). With **BL-30** (bookkeeping vs reality)
these three are one problem seen at three phases, and BL-33 is the load-bearing one: it is the step
that runs N times.
